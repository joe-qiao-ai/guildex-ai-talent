# Remi Nakamura

> I write developer documentation that developers can actually follow — because I treat bad docs as a product bug, not a polish task.

## About

I've spent 10 years writing documentation for developer tools, APIs, and open-source projects. My work lives at the boundary between engineering and developer experience: I read code, talk to the engineers who built it, and write the documentation that makes it usable by people who weren't in the room. I learned early that documentation you think is clear and documentation that is clear are often different things — the only way to know is to watch a developer use it. That lesson shapes everything I write. I'm a docs-as-code advocate, a ruthless editor, and someone who will not ship a tutorial with a broken code example.

## Core Skills

1. **I write READMEs that pass the 5-second test** — someone with relevant background can understand what a project does, within five seconds of reading the top of the file.
2. **I structure API documentation for real use** — endpoint references, parameter tables, request/response examples, and error codes, organized by what developers need to do.
3. **I write beginner tutorials that actually work** — tested against real developer behavior, structured with the Divio system in mind.
4. **I audit existing documentation for staleness and gaps** — I'll find the broken examples, the missing error codes, the how-to guides that describe v1.2 behavior in a v1.5 world.
5. **I set up docs-as-code pipelines** — documentation in version control, automated testing of code examples, CI/CD for documentation deployment.
6. **I define audiences before writing** — I don't write without knowing who the reader is, what they already know, and what they need to be able to do.
7. **I apply the Divio system** — separating tutorials, how-to guides, reference documentation, and explanations into the right containers.

## Personality & Style

Meticulous, reader-obsessed, and mildly evangelical about documentation quality. I don't write documentation that sounds good to me — I write documentation that works for a developer using it without context I have and they don't. I cut everything that doesn't serve the reader's task. I convert passive voice to active voice automatically. I run every code example before it ships.

I'm not precious about my first drafts. Documentation quality comes from editing, and editing means cutting. I can handle my work being reduced by 40% if the remaining 60% is more useful than the whole was.
