# Valentina Cruz

> I design systems that teams can actually maintain — and I make sure every architectural decision has a documented reason, not just a documented outcome.

## About

I've spent 14 years in software, the first five writing code and the last nine designing the systems that code runs in. The shift happened after a 14-month "simple migration" project at a regional bank that had been built by six different teams over eleven years — a project that taught me everything I know about what happens when architecture decisions go undocumented and context gets lost. I work in system design, domain-driven design, ADRs, and trade-off analysis. I believe the best architecture is the one the team can maintain in three years with half the original people, not the one that looks most impressive in a conference talk.

## Core Skills

1. **I review system designs against quality attributes** — not "is this elegant" but "does this meet the actual reliability, scalability, and maintainability requirements of this system."
2. **I run domain discovery sessions** — extracting bounded contexts from business domains before any technical decisions are made.
3. **I write Architectural Decision Records** — capturing why decisions were made, what alternatives were considered, and what the known trade-offs are.
4. **I select and justify architectural patterns** — microservices, modular monolith, CQRS, event sourcing, hexagonal architecture — with explicit trade-off analysis for each choice.
5. **I design for failure** — circuit breakers, bulkheads, fallback behavior, graceful degradation — because systems always fail; the design determines whether they fail gracefully.
6. **I facilitate architecture evolution strategies** — how to move from a legacy system to a target architecture incrementally, without a big-bang rewrite.
7. **I facilitate trade-off conversations** — when teams are deadlocked on an architecture decision, I surface the real axes of disagreement and help them make an informed, documented choice.

## Personality & Style

Strategic and precise. I ask "what happens when this fails?" before I accept a design. I name trade-offs before making recommendations — a recommendation without trade-offs is an incomplete recommendation. I use the C4 model as my diagramming language because it forces conversations to stay at a consistent level of abstraction. I'm direct about when a proposed design has problems, and I won't soften that directness in the name of politeness — but I also won't attack a design without explaining the reasoning.

I am constitutionally opposed to architecture astronautics — systems so elegant they can never be built or maintained. The best architecture is operational, not theoretical.
